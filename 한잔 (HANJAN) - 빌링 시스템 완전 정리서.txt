HANJAN 빌링 시스템 완전 정리서
크리스로님, 지금까지 논의된 빌링·수익 관련 내용을 하나도 빠짐없이 체계적으로 정리해드리겠습니다.

1. 수익 모델 아키텍처
1-1. 모델 진화 과정
초기 기획에서는 O2O 제휴 수수료 + 프리미엄 구독 + 브랜드 스폰서 3개 레이어를 계획했으나, 크리스로님의 결정으로 F6(파트너 장소)를 Phase 2로 연기하면서 현재 MVP에서는 프리미엄 구독 + 인앱 단건 구매 2개 레이어 하이브리드 모델로 전환되었습니다.

1-2. 현재 MVP 수익 구조 (2-Layer Hybrid)
Layer 1 — HANJAN Plus 구독 (정기 결제)

월간 플랜은 ₩9,900/월이며, 연간 플랜은 ₩59,900/년(월 환산 ₩4,991, 약 50% 할인)입니다. Plus 구독자에게 제공되는 혜택은 다음과 같습니다: 무제한 활동 생성(무료 회원은 2회/일), 무제한 매칭 수락(무료 회원은 3회/일), 매칭 요청 미리보기(상대가 나를 선택하기 전에 확인), 매너 점수 상세 리포트, 활동 카드 부스트(상단 노출 우선), 타이머 채팅 45분(무료 회원은 30분), Plus 배지 표시, 고급 매칭 필터(직업군, 연령대 세분화, 관심사 상세 등)가 포함됩니다.

Layer 2 — 인앱 단건 구매 (IAP)

네 가지 상품이 있습니다. "지금 부스트"는 ₩2,900으로 활동 카드를 1시간 동안 상단에 노출시켜 매칭 확률을 약 3배 높여줍니다. "한번 더"는 ₩1,900으로 하루 활동 생성 또는 매칭 수락 횟수를 1회 추가합니다. "타이머 연장"은 ₩1,500으로 진행 중인 타이머 채팅에 15분을 1회 추가합니다. "두번째 인상"은 ₩2,500으로 이전에 거절했던 상대에게 재매칭 요청을 보낼 수 있습니다.

1-3. Phase 2 확장 수익 (2027 Q1 예정)
F6 파트너 장소가 출시되면 3개 레이어가 추가됩니다. O2O 제휴 수수료로 쿠폰 사용당 ₩1,000~2,000, 월 고정 파트너십 ₩50,000~200,000이 예상되며, 브랜드 스폰서/광고로 네이티브 배너와 스폰서 이벤트 수익이 발생합니다.

2. 무료 vs 유료 기능 비교표
무료 회원의 경우, 활동 생성은 2회/일, 매칭 수락은 3회/일, 타이머 채팅은 30분, 안심 만남 기능은 무제한, 프로필 인증은 무제한, 매칭 미리보기와 부스트는 불가, 매칭 필터는 기본(반경·시간·인원)만 제공됩니다.

Plus 회원의 경우, 활동 생성과 매칭 수락은 무제한, 타이머 채팅은 45분, 안심 만남과 프로필 인증은 동일하게 무제한, 매칭 미리보기가 가능하고, 부스트는 주 1회 무료 제공, 매칭 필터는 고급(직업군·연령대·관심사 상세)까지 사용할 수 있습니다.

핵심 원칙은 "안전과 기본 매칭은 절대 유료화하지 않는다"입니다. 유료화되는 것은 편의성과 확장성뿐이며, 이는 사용자 신뢰와 리텐션을 지키기 위한 설계입니다.

3. 레퍼럴 리워드 빌링
3-1. 보상 규칙
기존 사용자가 3명의 친구를 초대하고, 그 3명 모두 회원 가입 + 본인 인증을 완료하면 즉시 Plus 1개월(30일) 무료가 부여됩니다. 무료 회원인 경우에는 30일간 Plus 기능이 활성화되고, 유료 회원인 경우에는 다음 결제 주기에서 1개월분 결제가 면제(₩9,900 크레딧 적용, 연간 플랜은 ₩4,991 환산)됩니다.

추가 3명을 또 달성할 때마다 1개월씩 누적되며 상한은 없습니다. 다만 동일 인물의 중복 추천은 불가합니다.

3-2. 사용자 경험(UX)
무료 회원이 리워드를 받으면 "Plus 무료 체험 중 — D-23일 남음" 형태의 배너가 표시되며, 만료 7일/3일/1일 전에 푸시 알림이 발송됩니다. 만료 시 자동으로 무료 티어로 복귀하며 자동 결제는 발생하지 않습니다. 유료 회원이 리워드를 받으면 "다음 결제 면제 — 2026.06.15까지 무료" 형태로 표시되고, 해당 기간 동안 정기 결제가 스킵됩니다.

3-3. 빌링 구현 방식 (3가지 옵션)
Option A — 인앱 크레딧 방식: 3명 달성 시 ₩9,900 크레딧을 사용자 계정에 적립하여 다음 결제 주기에 적용합니다. 장점은 정확한 금액 관리가 가능하지만, 환불 처리 로직이 필요하고 Apple/Google 정책과의 충돌 가능성이 있습니다.

Option B — 무료 기간 연장 방식 (MVP 추천): 현재 구독 종료일 이후에 30일 무료 기간을 추가합니다. 구현이 가장 단순하고 Apple/Google 정책 위반 리스크가 없으며, 무료/유료 회원 모두 동일한 로직으로 처리할 수 있습니다.

Option C — 하이브리드: Option B를 기본으로 사용하되, 무료/유료 구분 없이 동일하게 30일 무료 기간을 추가합니다. 로직을 최소화할 수 있습니다.

MVP에서는 Option B를 채택합니다. 이유는 RevenueCat의 promotional subscription 기능과 자연스럽게 연동되고, Apple의 "구독 할인처럼 보이는 보상 금지" 정책을 우회할 수 있으며("무료 프리미엄 기능 체험"으로 포지셔닝), 구현 복잡도가 가장 낮기 때문입니다.

3-4. 어뷰징 방지
본인 인증(신분증 + 휴대폰 번호) 필수, 동일 기기/전화번호 중복 차단, 추천받은 친구가 7일 이내 탈퇴 시 보상 자동 회수, IP 기반 대량 가입 모니터링, 비정상 패턴 감지 시 보상 일시 정지 후 수동 검토를 실시합니다.

3-5. 레퍼럴 ROI 시뮬레이션
초기 사용자 5,000명 기준으로, 30%(1,500명)가 레퍼럴을 시도하고 평균 5명씩 초대하면 총 초대 7,500건이 발생합니다. 12% 전환율을 적용하면 900명이 신규 가입하고, 이 중 600명의 referrer가 3명 달성 기준을 충족합니다. 보상 기회비용은 600 × ₩9,900 = ₩5,940,000이며, 레퍼럴 경유 CAC는 ₩5,940,000 ÷ 900 ≈ ₩6,600(일반 CAC ₩10,000~30,000 대비 크게 절감)입니다. 무료 체험 후 유료 전환율 25%를 적용하면 150명의 신규 유료 사용자가 생기고, 월 추가 매출 ₩1,485,000(연간 약 ₩17,820,000), ROI는 약 300%에 달합니다.

5. API 엔드포인트 전체 목록
5-1. 구독 관련
GET /api/subscriptions/status — 현재 구독 상태를 조회합니다. 응답에는 plan, status, currentPeriodStart, currentPeriodEnd, daysRemaining, isPlus(boolean), activeReferralReward(있을 경우) 정보가 포함됩니다.

GET /api/subscriptions/plus-access — Plus 접근 가능 여부를 확인하는 경량 엔드포인트입니다. 응답은 { hasAccess: boolean, source: "subscription" | "referral_reward" | "none", expiresAt: DateTime | null }입니다. 프론트엔드의 usePlusAccess 훅에서 호출하여 UI 토글(페이월 모달, Plus 배지, 기능 제한 해제)에 사용합니다.

POST /api/subscriptions/webhook — RevenueCat 웹훅 수신 엔드포인트입니다. 환경변수 REVENUECAT_WEBHOOK_SECRET으로 서명을 검증한 후, 이벤트 타입에 따라 DB의 Subscription 레코드를 업데이트합니다. 처리하는 이벤트는 INITIAL_PURCHASE(신규 구독 생성), RENEWAL(기간 갱신), CANCELLATION(취소 상태 변경), EXPIRATION(만료 처리), BILLING_ISSUE(결제 실패 알림)입니다.

5-2. 인앱 구매 관련
GET /api/purchases/products — 구매 가능한 인앱 상품 목록을 반환합니다. 각 상품의 productId, name, description, price, currency, type, duration(해당 시) 정보가 포함됩니다.

POST /api/purchases/verify — 클라이언트에서 전송한 영수증(receipt)을 RevenueCat API로 검증하고, 유효하면 Purchase 레코드를 생성합니다. 요청 바디는 { productId: string, revenuecatTransactionId: string, platform: "ios" | "android" }이며, 성공 시 Purchase 객체를 반환합니다.

GET /api/purchases/history — 사용자의 구매 이력을 최신순으로 반환합니다. 페이지네이션(page, limit)을 지원하며, 각 항목에 isConsumed, consumedAt 정보가 포함됩니다.

POST /api/purchases/consume — 소모성 상품(부스트, 타이머 연장 등)을 사용 처리합니다. 요청 바디는 { purchaseId: string }이며, isConsumed를 true로, consumedAt을 현재 시간으로 업데이트합니다.

5-3. 레퍼럴 관련
GET /api/referral/my-code — 사용자의 고유 레퍼럴 코드를 반환합니다. 없으면 자동 생성합니다.

GET /api/referral/status — 레퍼럴 현황을 반환합니다. 응답에는 totalInvited(총 초대 수), completedCount(가입 완료 수), pendingCount(대기 중), rewardsEarned(받은 보상 횟수), nextRewardProgress(다음 보상까지 n/3) 정보가 포함됩니다.

POST /api/referral/complete — 추천받은 사용자가 본인 인증을 완료했을 때 호출됩니다. 해당 Referral의 status를 COMPLETED로 변경하고, referrer의 completedCount가 3의 배수가 되면 자동으로 ReferralReward를 생성합니다. 보상 타입은 referrer의 구독 상태에 따라 결정됩니다: 활성 구독이 있으면 SKIP_PAYMENT, 없으면 FREE_PLUS.

GET /api/referral/reward-status — 현재 활성화된 레퍼럴 보상 정보를 반환합니다. 응답에는 rewardType, startDate, endDate, daysRemaining, isActive 정보가 포함됩니다.

Copy// hanjan-server/src/modules/subscriptions/subscriptions.controller.ts
@Post('webhook')
async handleWebhook(@Req() req: Request, @Res() res: Response) {
  const signature = req.headers['x-revenuecat-signature'];
  if (!this.verifySignature(signature, req.body)) {
    return res.status(401).send('Invalid signature');
  }

  const { event } = req.body;
  switch (event.type) {
    case 'INITIAL_PURCHASE':
      await this.subscriptionService.createSubscription(event);
      break;
    case 'RENEWAL':
      await this.subscriptionService.renewSubscription(event);
      break;
    case 'CANCELLATION':
      await this.subscriptionService.cancelSubscription(event);
      break;
    case 'EXPIRATION':
      await this.subscriptionService.expireSubscription(event);
      break;
    case 'BILLING_ISSUE':
      await this.subscriptionService.handleBillingIssue(event);
      break;
  }

  return res.status(200).send('OK');
}
6-5. Plus 접근 판정 로직 (백엔드)
Copy// hanjan-server/src/modules/subscriptions/subscriptions.service.ts
async checkPlusAccess(userId: string): Promise<PlusAccessResult> {
  // 1. 활성 구독 확인
  const subscription = await this.prisma.subscription.findUnique({
    where: { userId, status: 'ACTIVE' },
  });
  if (subscription && subscription.currentPeriodEnd > new Date()) {
    return { hasAccess: true, source: 'subscription', expiresAt: subscription.currentPeriodEnd };
  }

  // 2. 활성 레퍼럴 보상 확인
  const reward = await this.prisma.referralReward.findFirst({
    where: { referrerId: userId, isActive: true, endDate: { gt: new Date() } },
    orderBy: { endDate: 'desc' },
  });
  if (reward) {
    return { hasAccess: true, source: 'referral_reward', expiresAt: reward.endDate };
  }

  // 3. 접근 불가
  return { hasAccess: false, source: 'none', expiresAt: null };
}
이 로직이 핵심입니다. 구독이든 레퍼럴 보상이든, 하나라도 활성이면 Plus 기능을 열어주는 단일 판정 지점입니다.

7. 프론트엔드 빌링 훅 & 컴포넌트
7-1. 주요 훅
useSubscription() — /api/subscriptions/status를 폴링하여 현재 구독 상태를 반환합니다. 반환값: { plan, status, daysRemaining, isPlus, isLoading }.

usePlusAccess() — /api/subscriptions/plus-access를 호출하여 Plus 접근 가능 여부를 반환합니다. 반환값: { hasAccess, source, expiresAt, isLoading }. 이 훅 하나로 "이 사용자가 Plus인가?"를 앱 전역에서 판단합니다.

useReferralStatus() — /api/referral/status와 /api/referral/reward-status를 호출하여 레퍼럴 현황과 보상 상태를 반환합니다.

7-2. 주요 화면 (src/pages/subscription/)
index.tsx (구독 메인 화면) — 현재 플랜 표시, 무료/Plus 비교 표, 월간/연간 플랜 선택 카드, 구매 버튼. usePlusAccess를 사용하여 이미 Plus이면 "현재 플랜" 상태를, 아니면 업그레이드 CTA를 표시합니다.

purchase.tsx (구매 처리 화면) — RevenueCat SDK의 Purchases.purchasePackage()를 호출하고, 성공/실패/취소에 따른 UI를 표시합니다.

7-3. 페이월 모달
무료 회원이 Plus 전용 기능(예: 4번째 매칭 수락, 고급 필터)을 터치하면 자동으로 페이월 모달이 올라옵니다. 모달에는 해당 기능의 설명, Plus 혜택 요약, 가격, "Plus 시작하기" CTA, 그리고 "친구 3명 초대하면 1개월 무료!" 레퍼럴 유도 링크가 포함됩니다.

7-4. 레퍼럴 보상 UI
레퍼럴 허브 화면(src/pages/referral/)에는 초대 진행률 표시(예: "2/3명 완료 — 1명만 더!"), 보상 활성 시 남은 일수 배너("Plus 무료 체험 중 — D-23일"), 초대 링크 공유 버튼(카카오톡, 문자, 링크 복사), 초대 이력 리스트가 포함됩니다.

프로필/설정 화면에서는 현재 구독 상태와 함께, 레퍼럴 보상이 활성화되어 있으면 "레퍼럴 보상으로 Plus 이용 중 — ~2026.07.15"와 같이 만료일을 명확히 표시합니다.

8. 인앱 구매(IAP) 소모 로직
부스트, 한번 더, 타이머 연장, 두번째 인상은 모두 소모성 상품입니다. 구매 후 즉시 사용되거나, 사용자가 원하는 시점에 "사용"을 터치해야 합니다.

부스트: 구매 즉시 소모(isConsumed: true), expiresAt을 현재 시간 + 1시간으로 설정. 1시간 후 자동 만료.

한번 더: 구매 즉시 소모. 해당 일자의 활동 생성 또는 매칭 수락 limit을 +1 증가. 자정(KST)에 리셋.

타이머 연장: 진행 중인 채팅에서 "시간 연장" 버튼 터치 시 소모. 타이머에 15분 추가. 채팅당 1회만 사용 가능.

두번째 인상: 이전 매칭 목록에서 "다시 만나기" 버튼 터치 시 소모. 상대에게 재매칭 알림 발송.

9. 앱스토어 정책 준수 사항
Apple App Store
모든 디지털 상품(구독, IAP)은 Apple IAP를 통해서만 결제해야 합니다. RevenueCat이 이를 추상화하므로 직접 StoreKit을 다룰 필요는 없습니다. 레퍼럴 보상은 "구독 할인"이 아닌 "무료 프리미엄 기능 체험"으로 포지셔닝해야 Apple 정책 위반을 피할 수 있습니다. 자동 갱신 구독의 경우 구독 관리 딥링크(https://apps.apple.com/account/subscriptions)를 앱 내 설정에 제공해야 합니다. 소규모 개발자 프로그램(연 매출 $1M 미만) 적용 시 수수료는 15%입니다.

Google Play Store
동일하게 Google Play Billing을 통해 결제해야 합니다. RevenueCat이 처리합니다. 구독 해지 딥링크를 제공해야 합니다. 소규모 개발자(연 매출 $1M 미만)는 수수료 15%가 적용됩니다.

10. 수익 시뮬레이션
10-1. MVP 단계 (런칭 12개월 후, F6 없음)
가정: MAU 50,000, DAU 15,000, 구독 전환율 8%, 인앱 구매율 12%(DAU 대비 월 1회 이상 구매).

구독 매출은 50,000 × 8% = 4,000명 × ₩9,900 = ₩39,600,000/월입니다. (연간 플랜 비율을 고려하면 실제로는 약 ₩31,680,000~₩39,600,000 범위이며, 보수적으로 ₩31,680,000를 적용합니다.)

인앱 구매 매출은 15,000 × 12% = 1,800명/일 활성 구매자이지만, 월간 고유 구매자로 환산하면 약 6,000명이며, 평균 구매액 ₩5,200 적용 시 ₩31,200,000/월입니다.

총 월 예상 매출은 약 ₩62,880,000~₩70,800,000(약 ₩6,300~7,100만원)이며, 앱스토어 수수료 15% 차감 후 실수령은 약 ₩53,448,000~₩60,180,000/월입니다.

레퍼럴 보상 기회비용(월 약 ₩5,940,000)을 차감하면 순수익은 약 ₩47,500,000~₩54,240,000/월입니다.

10-2. Phase 2 (F6 파트너 장소 추가 후)
O2O 제휴 수수료 약 ₩80,000,000/월, 스폰서십 약 ₩15,000,000/월이 추가되어, 총 월 예상 매출은 약 ₩157,880,000~₩165,800,000(약 ₩1.58~1.66억)으로 증가합니다. O2O와 스폰서십은 앱스토어 수수료 대상이 아니므로 마진이 더 높습니다.

11. 핵심 KPI (빌링 관련)
추적해야 할 핵심 지표는 다음과 같습니다. MRR(Monthly Recurring Revenue)은 구독 기반 월 반복 매출입니다. ARPU(Average Revenue Per User)는 MAU 대비 월 매출로, 목표는 ₩1,400 이상입니다. 구독 전환율은 무료→유료 전환 비율로 목표 8~10%입니다. 월간 이탈률(Churn Rate)은 구독 해지율로 업계 평균 15~20%이며, 목표는 12% 이하입니다. LTV(Lifetime Value)는 사용자당 생애 가치로, 목표는 CAC의 3배 이상입니다. IAP 전환율은 DAU 중 인앱 구매 비율로, 목표 10~15%입니다. 레퍼럴 전환율은 초대받은 사람의 가입 완료 비율로, 목표 12% 이상입니다. 레퍼럴→유료 전환율은 무료 체험 후 유료 구독으로 전환하는 비율로, 목표 25%입니다.

14. 체크리스트
빌링 시스템 구현 전 확인해야 할 사항입니다.

RevenueCat 계정을 생성하고 프로젝트를 설정해야 합니다. Apple Developer 계정에서 App Store Connect의 구독 상품(월간/연간)과 소모성 IAP 4종을 등록해야 합니다. Google Play Console에서도 동일한 상품을 등록해야 합니다. RevenueCat 대시보드에서 해당 상품들을 Entitlements와 Offerings에 매핑해야 합니다. 웹훅 URL을 RevenueCat에 등록하고 시크릿 키를 서버 환경변수에 설정해야 합니다. 샌드박스 환경에서 구독 구매, 갱신, 취소, 만료 전체 플로우를 테스트해야 합니다. 레퍼럴 보상의 경우, 3명 달성 → 보상 생성 → Plus 활성화 → 만료 복귀 전체 시나리오를 테스트해야 합니다.
